01 How to build this

Read this first

The HTML files in this handoff are design references — prototypes showing the intended look and behaviour, not production code to copy. Your job is to recreate them in the target environment (React Native / Expo is the assumed stack) using its established patterns, then wire them to the local data layer.

  1. Recreate, don't transplant. Rebuild each screen with native components (View/Text/Pressable, FlatList, Reanimated). The HTML/React prototype demonstrates layout, hierarchy and motion — match it pixel-for-pixel, but in idiomatic platform code.
  2. Tokens are the contract. Port the values in §03 into constants/{colors,typography,layout}.ts and reference them everywhere — never raw hex in product code.
  3. Build the component library first (§04), then compose screens from it. Every screen in §05 is assembled from these 17 primitives — no bespoke one-off layouts.
  4. Money is integer paise end-to-end. Parse and store as paise (₹1 = 100); format only at the view layer. Never use floats for money.
  5. Match the interaction model (§06). The donut, the trend chart and the budget-impact preview are the product's signature moments — get the tap-to-reveal behaviour right, not just the static look.
Open the BudgetSplit App tab and click through every flow before you start — the spec describes behaviour, but the prototype is the behaviour.
02 Design principles

Five rules that hold the app together

  • One hero per screen. Each screen opens with a single dominant element — a spending figure, a net balance, a trend. Everything else is secondary and quieter.
  • Progressive disclosure. Show the summary; reveal detail on tap. The donut, trend bars and category rows all start collapsed and expand into specifics — never a wall of numbers up front.
  • Money has its own voice. Every amount is Space Mono, sign-coloured (green income / coral expense), and never decorative.
  • Cards group, hairlines divide. Content lives in bordered cards on a near-black canvas; depth comes from soft black halos, not heavy shadows.
  • Motion confirms, never decorates. Every animation answers “did my tap register?” or “where did this come from?” No idle loops, no parallax.

Typography

Two families, strict roles: Inter for every word, Space Mono for every money amount. Max three sizes per screen.

  • Title — 28 / 600 / -0.4px · screen headers
  • Heading — 20 / 600 · pushed-screen titles
  • Body — 15 / 400 · all running copy
  • Caption — 12–13 / 400 · meta, muted
  • Amount — Space Mono, 18–44 / -0.5px · hero figures

Spacing, radius & elevation

  • Spacing — 4 · 8 · 16 · 24 · 32 · 48. Card padding 16–22; gutters 16.
  • Radius — inputs/buttons 12, cards 16–20, pills 999.
  • Elevation — shadow-sm (cards/rows) → shadow-md (hero card) → shadow-lg (sheets/modals). Soft black halos, not hard drop-shadows.
  • FAB — teal→coral gradient with its own coral glow.
  • Hairline — 1px #21302E border between grouped rows.

06 Signature interactions

The three moments that define the app

These tap-to-reveal interactions are the product’s personality. Implement the behaviour, not just the static frame.

Category donut

Idle: ring + centre total + “tap a slice”. Tap a wedge → it translates outward ~7px and grows, siblings drop to 0.32 opacity, centre cross-fades to the category’s name + amount + “{pct}% · View →”. Tap centre or wedge again → drill into Category Detail. Chips below mirror the selection. Easing: spring cubic-bezier(0.34,1.56,0.64,1).

Trend chart

Bars sized to value against a dashed budget line; over-budget bars are amber. Tap a bar → lift + glow, siblings dim to 0.3, a detail card fades in with the period total, an over/under Badge, and tappable top-category rows. Re-tap to deselect. Month/Year switch animates bar heights and clears selection.

Budget impact

In Add Expense, choosing a budgeted category reveals a live card: the projected spend vs allocation, a BudgetBar at the new level, and curPct → newPct in health colour. The user sees the consequence of the expense before saving — entry meets analytics.

08 Feature roadmap

What to build next — ranked

In-scope extensions of the current model — no backend, no AI. Build tier 1 first: each reuses existing components and removes a real daily friction.

Tier 1 · build first high impact · low effort

Recurring auto-post

Auto-create transactions on their due date with a “review” nudge. Recurring lines & cadence already modelled. Reuses TransactionRow + Badge.

CSV / PDF export

One “Export” row in Settings → share sheet. Pure local read of the data — no new model. Offline users want portability.

Smart settle-up

Debt simplification — minimise the number of payments to square a group. Logic only; the “Who paid what” + Settle UI already exist.

Tier 2 · strong follow-ups med effort

Budget rollover & alerts

Carry unspent budget to next cadence (rollover rules already shown per group); local notification at 80% / 100% utilisation.

Encrypted local backup

Export an encrypted file to the user’s own storage & restore — protects the no-cloud promise against phone changes.

Search across transactions

A search field on History over note / category / amount. High everyday utility, small surface.

Tier 3 · extend the new analytics grows the redesign

Trend ↔ donut linking

Tapping a trend period filters the donut to that week — one continuous analytic story across Reports and the dashboard.

“Safe to spend” chip

Daily allowance from remaining budget ÷ days left, shown on the dashboard hero — turns the budget-impact insight into forward guidance.

Member nudges

From “Who paid what”, a one-tap reminder to whoever owes the group most — closes the settle loop.

09 Data model

Local, offline, integer paise

On-device SQLite, no backend. Money is always integer paise. The prototype’s data.js mirrors these shapes — use it as the seed contract.

  • person — name, colour, (email, mobile optional)
  • group — id, name, icon, colour, members, health, allocated/spent
  • txn — kind (expense / income / transfer / settlement), category, note, icon, colour, paise, when, (tz, lat/lng/place optional)
  • budget_line — name, icon, colour, allocated, spent, cadence
  • category — name, icon, colour (master catalog)
  • balance — per person net paise (− you owe, + owes you)
  • reports — trend series (bars w/ top categories) + budget overlay
Format only at the view layer (formatRupees / formatRupeesShort). Wrap multi-table writes in a transaction. Icons are Feather (2px stroke); Inter + Space Mono load from Google Fonts — self-host woff2 for true offline parity.
