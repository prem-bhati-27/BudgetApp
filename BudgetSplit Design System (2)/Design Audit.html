01 Repository Design Audit

What the codebase actually is

BudgetSplit is a single offline-first Expo / React Native app — SQLite on-device, no backend, no accounts, no cloud. It does two jobs: budget your own money and split shared bills. Everything below is inventoried from the repo, not assumed.

Architecture

  • Routing — Expo Router (file-based). A (tabs) group for the four destinations, plus stacked routes under add/, group/[id]/, and txn/[id].
  • Data — local SQLite via expo-sqlite. Multi-table writes wrapped in withTransactionAsync(). Money stored as integer paise, never floats.
  • Domain logic — src/lib/: money.ts (parse/format ₹), analytics.ts (utilization, at-risk, projection, recommendations), settle-up minimization.
  • Component buckets — ui/ (generic), finance/ (domain), system/ (onboarding, gates). Strict dependency direction: ui never imports domain.
  • Data models — person (name, email, mobile), group, txn (kind: expense / income / transfer / settlement; tz + optional lat/lng/place), budget_line (cadence), category.
  • Privacy — biometric lock gate + app-switcher blur on backgrounding. Optional, user-controlled transaction location via expo-location.
  • Platform — iOS + Android, portrait phone only. Safe-area aware; tab bar height tracks bottom insets.
  • Design contract — AGENTS.md is a strict, non-negotiable design spec; this audit treats it as ground truth.

Existing design & UX patterns (observed, not prescribed)

One hero element per screen · everything grouped in bordered cards · settings-style icon-circle rows · Space Mono for all money · Feather icons at 2px in tinted circles · sectioned transaction lists (Today / Yesterday / date) · bottom sheets for add flows · segmented pills for periods · animated budget bars · empty states with icon-circle + title + body + CTA · spring-scale press feedback · haptics only on meaningful actions.

What's working: the visual system is unusually disciplined for an indie app — consistent cards, one hero per screen, money typography, and real empty/loading states are already specified. Top UX risks to manage

Density

Itemized split & group budget pack many controls — protect the hero number and use progressive disclosure (cadence behind a sheet).

Reachability

Long forms can push primary actions under the keyboard. Dynamic bottom padding keeps Save reachable.

Destructive

Removing budget lines / members needs confirmation + haptic warning; today some are one-tap.

State matrix

The nine required states, defined once and applied consistently:

Default

Token colors at rest; cards on bg-card + hairline + shadow-sm.

Hover / Press

Spring scale to 0.97 (FAB 0.9) — the motion is the feedback, not a color shift.

Active / Selected

Solid teal fill + dark text (pills, tabs); 2px white ring (avatars); teal border (options).

Focus

Input border switches to border-focus teal; visible ring for keyboard nav.

Disabled

40% opacity, pointer events off; never a different hue.

Loading

Skeleton blocks on the surface — never a bare full-screen spinner. Buttons show inline progress.

Error

Coral border + helper text; haptic error on validation failure.

Success

Income/health green, brief "Saved" confirm + haptic success.

Empty

Mandatory EmptyState layout — icon-circle, title, explanation, primary CTA.

05 Asset Inventory

A consistent visual language

BudgetSplit is deliberately illustration-light — meaning is carried by Feather icons in tinted circles, color-coded money, and charts. The only brand image is the donut ring mark.

Brand & iconography

  • Logo — multi-segment donut/ring in the four brand hues on near-black. Files: logo-icon.png (tile), logo-mark.png (transparent), splash-icon.png, favicon.png.
  • Icons — Feather only, uniform 2px stroke. No emoji, no Unicode glyphs. Loaded via the official Feather web build in this system.
  • Icon circles — the signature motif: glyph centered in a circle of its own color at ~13% opacity. Sizes 64 / 40 / 36 / 32.

Status & data graphics

  • Category donut — conic spend breakdown with center count + legend (amount + %).
  • Trend chart — weekly/monthly bars or area with readable axes & value-on-tap.
  • Budget bars — animated health fill (green / amber / red) as the at-a-glance status graphic.
  • Empty-state graphics — the icon-circle (no bespoke illustration needed), recolored per context.
One substitution flagged: the native app ships Feather as a vector font; this web system uses the official Feather web build (identical glyphs & names). Inter + Space Mono load from Google Fonts — swap to self-hosted woff2 for production if offline parity is required.

08 Responsive Specifications

Phone-first, safe-area aware

The product is a portrait phone app — responsiveness here means adapting to device width, notch/inset, dynamic type and the keyboard, not desktop breakpoints.

Layout rules

  • Single fluid column, 16px gutters; cards stretch full width.
  • Range 320–430pt wide. Hero number scales down one step below 360pt to avoid wrap.
  • Safe-area insets honored top & bottom; tab bar = base height + bottom inset.
  • Category & member grids reflow column count by available width (auto-fit).

Keyboard & input

  • Auto-focus the amount field; numeric keypad first on money entry.
  • Smooth-scroll the focused field into view; dynamic bottom padding keeps Save above the keyboard.
  • Respects OS Dynamic Type — the 3-size scale leaves headroom to grow.
  • Portrait-locked; no landscape layout required by the product.
09 Accessibility Review

Usable, with two things to watch

The system starts strong: 44pt touch targets are mandated, text contrast on the dark surfaces is high, and money is never color-only when a sign/label is present. Two areas need attention.

AreaStatusNote
Touch targets ≥ 44ptPassMandated in AGENTS.md; rows ≥ 52pt, icon buttons use hitSlop.
Primary text contrastPass#ECF3F1 on #0A0F11 ≈ 16:1.
Muted text contrastWatchtext-muted on bg ≈ 4.0:1 — fine for captions, avoid for essential body.
Color-only meaningWatchBudget health is green/amber/red — pair the bar with a label or icon for color-blind users.
Sign vs color for moneyPassAmounts carry +/− and category context, not just hue.
Reduced motionPassEntrances are additive over a visible end-state.
Screen-reader labelsWatchEnsure icon-only buttons (FAB, chevrons) have accessibility labels.
10 Implementation Recommendations

From this system into the app

  • Keep the token source single. constants/{colors,typography,layout}.ts ↔ this styles.css are 1:1 — generate one from the other to prevent drift.
  • Lock V2 screens to the spec. Itemized, transfer, settle, group budget and help should reuse Card / SettingsRow / AmountText / EmptyState exactly — no bespoke layouts.
  • Money stays integer paise end-to-end; format only at the view via formatRupees.
  • Confirm destructive actions (remove budget line / member) with a dialog + haptic warning (Phase C2).
  • Defer heavy analytics off the boot path; lazy-load charts (Phase E1) so first paint stays instant.
  • Add a11y labels to icon-only controls and a non-color budget-health indicator.
  • Use this repo as the contract — components, prompts and cards are the shared language for design ↔ eng handoff.
11 Feature Suggestions · 80/20

High-leverage, low-effort, in-scope

Strictly extensions of what already exists — no AI, no scope creep. Ranked by the 80/20 rule: most value for least build, reusing components and the local data model you already have.

High impact · Low effort

1 · Recurring auto-post

You already store recurring lines & cadence. Auto-create the transaction on its due date with a "review" nudge — turns manual logging into set-and-forget. Reuses TransactionRow + Badge.

High impact · Low effort

2 · CSV / PDF export

Offline-first users want their data portable. One "Export" row in Settings → share sheet. Pure local read of the SQLite tables; no new model.

High impact · Low effort

3 · Smart settle-up

Minimize the number of payments across a group (debt simplification) on the existing settle screen — fewer transactions to square up. Logic only; UI already there.

High impact · Med effort

4 · Budget rollover & alerts

Carry unspent budget to next cadence; local notification at 80% / 100% utilization (your analytics engine already computes both). Reuses BudgetBar + Badge.

Solid · Med effort

5 · Encrypted local backup & restore

No-cloud users fear losing data on phone change. Export an encrypted file to their own storage and restore it — keeps the privacy promise intact.

Nice · Med effort

6 · Search across transactions

A search field on History (FilterBar already exists) over note / category / amount. High everyday utility, small surface.

The 20% to build first: Recurring auto-post, CSV/PDF export, and smart settle-up. Together they reuse existing components and the local data model, ship without a backend, and remove the three biggest day-to-day frictions — manual entry, data lock-in, and settle math.
